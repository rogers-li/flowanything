# 智能语音交互机器人项目技术调研

## 1. 文档目标

本文档用于对“智能语音交互机器人”项目进行技术调研和方案梳理，重点覆盖以下关注点：

- 语音机器人与文本机器人的架构关系
- 文本智能核心是否可复用，以及如何拆分团队边界
- 语音交互相对文本交互带来的架构挑战
- 会话管理、状态机和任务管理的设计方式
- NLP、LLM、小模型/重模型混合路由的必要性与落地方式
- 语音场景下如何评估效果，以及如何持续迭代优化

本文档以“可落地的工程架构”而不是“单一模型能力”作为主要讨论对象。

---

## 2. 项目背景与目标

目标产品是一个支持多语言、多国家部署的智能语音交互机器人，既能通过语音回答用户问题，也能接入 AI Agent、Tools、Skills 等能力，帮助用户完成非语言类任务，例如信息查询、业务办理、任务执行和跨系统操作。

项目目标包括：

- 支持语音问答与多轮对话
- 支持 Agent 驱动的任务执行
- 支持多语言与多地区本地化
- 支持实时交互、快速响应与用户打断
- 支持复用统一的文本智能核心
- 支持后续扩展到文本、电话、车机、IM 等渠道

---

## 3. 核心结论

### 3.1 语音机器人不是“文本机器人 + 简单 ASR/TTS 适配器”

可以复用文本智能核心，但语音机器人并不只是给文本机器人加“耳朵和嘴巴”。真正可用的语音机器人还需要新增：

- 实时语音链路
- VAD / endpointing / barge-in
- 交互状态机
- 增量响应与口播优化
- 会话自动归并、挂起和恢复
- 面向语音的呈现与节奏控制

更准确的理解应为：

`语音机器人 = 文本智能核心 + 语音交互运行时 + 实时流式编排层`

### 3.2 文本智能核心可以高度复用，但前提是抽象正确

如果“文本机器人”被设计成：

- 输入是一段文本
- 输出是一段文本

则复用性会较差。

如果“文本机器人”被设计成：

- 输入是标准化事件
- 内部维护统一任务状态机
- 输出是动作计划和结构化结果

则它可以很好复用于语音机器人、电话机器人、企业 IM 机器人等。

### 3.3 最合理的组织方式是“两大团队 + 统一接口契约”

建议拆分为：

- 文本智能核心团队
- 语音交互团队

如团队规模允许，可增加平台/基础设施团队。

前提是两个团队要尽早冻结以下契约：

- Event Schema
- Action Schema
- Task State Schema
- Tool Invocation Schema
- Error Schema

## 4. 技术难点概览

### 4.1 实时性与低延迟

语音交互对首响应时间极其敏感。系统需要同时处理：

- 实时音频接入
- 流式 ASR
- 流式或增量式 LLM
- 流式 TTS
- 用户打断和播放停止

语音产品通常需要同时优化：

- 首字响应时间
- 最终结果返回时间
- TTS 首包时间
- 工具执行期间的占位反馈

### 4.2 多语言与多国家

挑战包括：

- 多语言 ASR、TTS、LLM 表现不均衡
- 口音、夹杂表达、中英混说
- 时间、日期、货币、地址、姓名格式本地化
- 数据合规、跨境传输与区域部署要求

### 4.3 对话管理复杂度高于文本

语音对话中用户会：

- 边想边说
- 频繁停顿和自我修正
- 使用更口语化、更省略的表达
- 在系统播报时打断

因此系统不仅要理解内容，还要管理回合、等待、抢答、中断和恢复。

### 4.4 Agent 执行安全

一旦机器人开始真正执行任务，而不仅仅是回答问题，就会面临：

- 工具误调用
- 高风险操作确认
- 权限越界
- 多步工作流失败恢复
- 审计与回放

### 4.5 评测对象是“整条链路”而不是单一模型

很多问题并不是 LLM 本身导致，而是：

- VAD 误判
- ASR 将关键实体识别错误
- 路由失误
- 工具参数错误
- TTS 口播不自然

因此评估必须是端到端的。

## 5. 整体架构建议

### 5.1 高层架构

建议整体架构分为六层：

1. 客户端/设备层
2. 实时语音接入层
3. 智能对话与编排层
4. Agent / Tools / Workflow 层
5. 业务系统接入层
6. 平台治理与观测层

### 5.2 参考链路

```text
Mic / Speaker / UI
    ->
Client Audio Stack
    ->
Realtime Gateway
    ->
ASR / VAD / Endpointing
    ->
Session Manager
    ->
Core Conversation Runtime
    ->
Agent / Tools / RAG / Workflow
    ->
Presentation Layer
    ->
TTS / Text Rendering
    ->
Audio / Screen Output
```

### 5.3 各层职责

#### 5.3.1 客户端/设备层

负责：

- 音频采集与播放
- 降噪、回声消除、唤醒词
- 客户端 VAD
- barge-in 本地检测
- 设备和网络状态管理

#### 5.3.2 实时语音接入层

负责：

- WebSocket / WebRTC 长连接
- 音频流转发
- 会话连接管理
- 鉴权、限流、路由
- 多端接入统一化

#### 5.3.3 智能对话与编排层

负责：

- 会话解析
- 上下文加载
- 任务状态推进
- LLM 推理和策略控制
- 动作规划

#### 5.3.4 Agent / Tools / Workflow 层

负责：

- 工具调用
- 多步任务编排
- 重试、超时、幂等
- 高风险操作确认
- 执行日志和审计

#### 5.3.5 业务系统接入层

负责对接：

- CRM / ERP / 工单
- 日历、邮件、IM
- 知识库、检索系统
- 地图、天气、物流、订单等外部服务

#### 5.3.6 平台治理与观测层

负责：

- 日志与链路追踪
- 评测集与回放
- Prompt / 策略版本管理
- 灰度发布与 A/B 测试
- 权限、合规、安全审计

## 6. 文本智能核心的可复用架构

### 6.1 设计目标

将文本机器人设计成一个“渠道无关的智能运行时”，而不是“聊天框后端”。

其核心抽象应为：

`Event -> Runtime -> Action`

而不是：

`Text -> LLM -> Text`

### 6.2 核心分层

建议拆为五层：

1. Channel Adapter
2. Conversation Runtime
3. Reasoning & Policy
4. Execution Layer
5. Memory & State

### 6.3 三个关键抽象

#### 6.3.1 Event

统一输入事件，而不是只接收文本。

示例事件类型：

- `user_message_committed`
- `user_utterance_committed`
- `user_interrupt_detected`
- `tool_result_received`
- `tool_failed`
- `timeout_occurred`
- `conversation_reset_requested`
- `session_resumed`

示例：

```json
{
  "event_id": "evt_001",
  "type": "user_utterance_committed",
  "session_id": "s_123",
  "user_id": "u_456",
  "channel": "voice",
  "locale": "zh-CN",
  "payload": {
    "text": "帮我查一下明天下午去东京的航班",
    "language": "zh-CN",
    "confidence": 0.94
  }
}
```

#### 6.3.2 Task State

统一任务状态机，跨文本和语音复用。

建议任务状态包括：

- `idle`
- `understanding`
- `awaiting_slot`
- `awaiting_confirmation`
- `executing_tool`
- `awaiting_tool_result`
- `responding`
- `completed`
- `failed`

示例：

```json
{
  "task_id": "task_789",
  "intent": "flight_search",
  "status": "awaiting_slot",
  "slots": {
    "from_city": "上海",
    "to_city": "东京",
    "date": "2026-04-25",
    "time_range": "afternoon"
  },
  "missing_slots": ["departure_airport"]
}
```

#### 6.3.3 Action

核心层输出不应只是 `reply_text`，而应是一组动作。

建议动作包括：

- `speak`
- `display_text`
- `ask_question`
- `ask_confirmation`
- `call_tool`
- `handoff_human`
- `wait`
- `end_turn`

示例：

```json
{
  "session_id": "s_123",
  "actions": [
    {
      "type": "speak",
      "text": "好的，我先帮你查一下。"
    },
    {
      "type": "call_tool",
      "tool": "search_flights",
      "args": {
        "from_city": "上海",
        "to_city": "东京",
        "date": "2026-04-25",
        "time_range": "afternoon"
      }
    }
  ],
  "state_patch": {
    "task_status": "awaiting_tool_result"
  }
}
```

### 6.4 为什么这种抽象适合语音复用

因为语音和文本的主要差异，不应落在核心的任务理解和执行能力上，而应落在：

- 渠道输入事件来源不同
- 输出动作的呈现方式不同
- 交互状态机和节奏不同

这意味着核心的：

- Intent 理解
- Slot 抽取
- Agent 调度
- Tool 调用
- Task State
- RAG
- 权限与审计

都可以复用。

## 7. 语音机器人新增能力与运行时设计

### 7.1 语音专属新增层

在复用文本核心的基础上，语音需要新增：

- 实时音频链路
- VAD
- ASR endpointing
- TTS 播放控制
- barge-in
- 交互状态机
- 口播优化
- 会话自动关闭与恢复

### 7.2 VAD 放在哪一层

推荐采用双层 VAD：

- 客户端 VAD 作为主触发器
- 服务端 VAD 作为兜底和统一校正

原因：

- 客户端离麦克风近，延迟最低
- 可更早判断用户开始说话
- 更适合处理用户打断
- 服务端更适合统一策略和多端一致性

最佳实践通常是：

- 客户端 VAD：快速触发和打断
- 服务端 VAD：兜底和统一校验
- ASR endpointing：判断一句话是否真正结束

### 7.3 流式 TTS 的实现建议

存在两种常见方式：

1. 固定文本输入，流式音频输出
2. 流式文本输入，增量语音输出

更成熟稳妥的工程实践是：

- LLM 流式输出 token
- 文本聚合器按短句或意群切块
- 每个稳定 chunk 再送入 TTS
- TTS 逐块流式合成和播放

推荐路线：

- 第一阶段：完整句子输入 + 流式音频输出
- 第二阶段：短句级增量 TTS
- 第三阶段：近实时边说边生成

### 7.4 语音交互不是简单的文本+语音转换

文本机器人通常是离散回合。

语音机器人则是连续流处理，需要新增：

- turn-taking
- 抢答/等待策略
- 中断处理
- 增量语义理解
- 口播压缩和摘要
- 工具执行期间的中间态反馈

这使得语音项目更接近“实时交互系统”，而不是传统聊天后端。

## 8. 文本与语音的产品体验差异对架构的影响

### 8.1 核心差异

文本交互适合：

- 长内容
- 多次输出
- plan、trace、tool 状态可视化
- 列表、表格、链接、细节展开

语音交互适合：

- 快速首响应
- 短句播报
- 渐进式反馈
- 摘要优先
- 必要时追问
- 复杂信息转屏幕展示

### 8.2 对架构的约束

这意味着核心层不能只返回最终显示文本，而应返回：

- 决策结果
- 结构化结果
- 动作序列
- 面向不同渠道的呈现提示

### 8.3 推荐增加 Presentation Layer

建议在核心 Runtime 外增加 Presentation / Response Adaptation Layer。

职责：

- 将核心动作投影到不同渠道
- 控制信息密度与表达长度
- 对语音进行口语化改写和摘要
- 将复杂结果拆成 summary 和 details
- 决定哪些内容“播报”，哪些内容“仅显示”

建议核心输出天然分层：

- `summary`
- `details`
- `next_choices`
- `structured_result`

这会极大提升文本和语音跨渠道复用能力。

## 9. 会话管理设计

### 9.1 语音会话比文本会话更复杂

文本中，用户可以显式点击“新建会话”。

语音中，系统通常只能感知：

- 用户是否正在说话
- 用户是否沉默
- 当前是否存在活跃任务
- 新输入是否和旧任务相关

因此语音会话必须通过状态和规则自动管理。

### 9.2 推荐拆成三层会话

#### 9.2.1 连接会话

表示一次设备与服务端的连接，例如一次 WebSocket / WebRTC 会话。

#### 9.2.2 交互会话

表示一段连续对话窗口，用于承载实时交互体验。

#### 9.2.3 任务会话

表示某个具体任务的生命周期，可跨多轮存在，甚至跨交互会话恢复。

关键原则：

- 连接会话可以很短
- 交互会话自动开始和关闭
- 任务会话可以长期保留

### 9.3 交互状态建议

建议交互状态机至少包括：

- `idle`
- `listening`
- `engaged`
- `waiting_user`
- `executing_task`
- `suspended`
- `closed`

### 9.4 会话开始与结束

开始触发方式通常包括：

- 唤醒词
- 按键/点击说话
- 持续监听 + VAD

结束触发方式通常包括：

- 静默超时
- 任务完成
- 明确结束语
- 设备/场景切换

### 9.5 会话管理核心原则

建议分离：

- 短期交互会话
- 长期任务上下文

例如用户沉默 40 秒后，交互会话可以关闭或挂起，但任务状态不应立即丢失。

用户稍后再说“最便宜的那个”，系统应能正确恢复任务上下文。

### 9.6 推荐新增 Session Manager

建议增加专门的 `Conversation Session Manager`，职责包括：

- 创建/关闭交互会话
- 超时管理
- 确定新话语归属哪个会话/任务
- 挂起和恢复任务
- 处理“重新开始”“换个话题”等会话控制意图

其职责边界为：

- Session Manager 决定“这句话属于谁”
- Core Runtime 决定“接下来做什么”

### 9.7 一个务实的 MVP 规则集

建议初期可使用如下规则：

1. 唤醒或按键说话且无活跃会话时，创建新交互会话
2. 30 秒内的新 utterance 默认归入当前交互会话
3. 若当前存在 `awaiting_confirmation` 或 `awaiting_slot`，优先续接当前任务
4. 超过交互超时则关闭交互会话，但保留任务状态 10 到 30 分钟
5. 用户说“重新开始/新问题/别管刚才的”时，新建交互会话并挂起旧任务
6. 若新输入和当前任务明显无关，则开启新任务或切换 active task

## 10. 混合模型设计：NLP、LLM 与 Agent

### 10.1 还要不要保留 NLP

有必要保留，但角色已经发生变化。

不建议再将传统 NLP 当作主大脑，但建议保留为：

- 前置路由层
- 低延迟路径
- 高确定性控制层
- 高风险操作守卫层

### 10.2 NLP/规则适合处理的场景

- 设备控制：暂停、停止、音量调节
- 会话控制：取消、重来、换个话题、重复
- 高风险确认：是/否、确认提交、删除
- 高频窄域场景：天气、物流、订单、余额
- 安全与治理：敏感词、越权请求、黑白名单

### 10.3 更适合交给 LLM 的场景

- 开放问答
- 跨领域自然表达
- 复杂多轮对话
- 模糊需求澄清
- 多约束解析
- 多工具任务规划

### 10.4 推荐的混合架构

```text
ASR Text
   ->
Fast Router (Rules / NLP / Small Model)
   ->
{ Direct Skill | Small LLM | Heavy LLM | Agent Workflow }
```

建议角色分层：

- NLP / 规则：做快、稳、确定的判断
- LLM：做泛化理解、复杂推理和自然表达
- Agent：做执行、编排和任务推进

## 11. 小模型与重模型的请求路由

### 11.1 核心问题

语音场景中，用户说了一句话之后，如何决定走小模型还是重模型。

最推荐的方法不是一次性猜准，而是做：

`小模型优先 + 低置信度/高复杂度时自动升级`

### 11.2 建议的 Router 设计

请求进入后先经过三层：

1. 规则层
2. 轻量分类层
3. 升级判定层

### 11.3 关键判断信号

#### 11.3.1 意图是否清晰

例如：

- “暂停播放”通常可走轻链路
- “帮我把刚才那个重新整理一下”通常需要重模型

#### 11.3.2 是否需要复杂推理

如果涉及：

- 多约束
- 多步规划
- 复杂上下文依赖
- 历史任务引用

则更适合重模型。

#### 11.3.3 工具调用复杂度

简单单工具调用可先走小模型。  
多工具组合、复杂参数补全则更适合重模型。

#### 11.3.4 风险等级

高风险操作建议默认走重模型并附带确认机制。

#### 11.3.5 用户话语复杂度

可以参考：

- 文本长度
- 约束条件数量
- 是否出现模糊指代
- 是否包含多个目标

#### 11.3.6 小模型自身置信度

若小模型对分类、槽位或动作判断置信度低，则自动升级。

### 11.4 语音场景中的推荐实践

语音里通常采用：

- 快速路由
- 先给一句占位反馈
- 后台重模型继续处理

即：

`fast acknowledge + heavy reasoning`

例如用户说：

“帮我订一张明天下午去东京的票，别太早，最好便宜一点。”

系统可先快速回复：

“好的，我来帮你看看。”

然后后台由重模型做理解、工具调用和结果筛选。

### 11.5 一套可落地的策略

#### 11.5.1 直接走小模型/规则

- 设备控制
- 会话控制
- 明确 yes/no
- 高频固定查询
- 单工具、单参数任务

#### 11.5.2 直接走重模型

- 开放问答
- 复杂任务
- 多轮依赖强
- 多工具规划
- 高风险操作
- 模糊表达

#### 11.5.3 小模型先试，失败再升级

- 中等复杂度查询
- 有结构但可能缺槽位的任务
- 表达模式不稳定但领域有限的请求

---

## 12. 如何确定特定场景下适用的 LLM

### 12.1 不是先挑模型，而是先定义场景要求

建议先按场景拆分，再定模型。典型场景包括：

- 实时交互类
- 复杂理解类
- Agent 执行类
- 结果改写类

### 12.2 每个场景关注的五个维度

- 质量
- 延迟
- 成本
- 稳定性
- 多语言能力

### 12.3 评测方式

建议每类场景构建小型 benchmark，至少覆盖：

- 用户原始话语
- 期望意图
- 期望槽位
- 期望动作
- 是否应调工具
- 是否适合语音播报

### 12.4 重点关注的任务级指标

- Intent Accuracy
- Slot Accuracy
- Action Accuracy
- Tool Call Success Rate
- Task Success Rate
- Voice Suitability

### 12.5 典型结论形式

最终很可能不是一个模型覆盖全部职责，而是：

- 模型 A：适合快速首响应
- 模型 B：适合复杂推理与 Agent
- 模型 C：适合改写和口播摘要

这会自然形成多模型协同。

## 13. 评估体系设计

### 13.1 总体原则

语音机器人必须按完整链路评估：

`听得清 -> 接得快 -> 理得准 -> 做得成 -> 说得自然`

建议从四层来评估：

- 语音层
- 对话层
- 任务层
- 体验层

### 13.2 语音层指标

输入侧：

- VAD 准确率
- 误触发率
- 截断率
- ASR 词错率
- 语言识别准确率

输出侧：

- TTS 首包时延
- 播放完成率
- 发音自然度
- 数字/日期/货币播报正确率
- 可懂度

### 13.3 对话层指标

- 首响应时间
- 最终响应时间
- 追问率
- 无效追问率
- 打断处理成功率
- 会话续接成功率
- 上下文理解准确率
- 确认策略正确率

### 13.4 任务层指标

- 意图识别准确率
- 槽位抽取准确率
- 工具调用成功率
- 任务完成率
- 平均轮次
- 转人工率
- 高风险确认命中率

### 13.5 体验层指标

- 用户满意度
- 重复表达率
- 放弃率
- 静默流失率
- 重新唤醒率
- 多语言满意度差异

### 13.6 诊断方式：漏斗分析

建议按以下漏斗分析每次交互：

1. 用户说话成功
2. VAD 正确截取
3. ASR 正确识别
4. 意图/槽位理解正确
5. 正确路由到模型/工具
6. 工具执行成功
7. 回复生成正确
8. TTS 播报自然
9. 用户继续互动
10. 任务完成

这有助于快速定位瓶颈所在。

## 14. 评测与优化闭环

### 14.1 离线评估集建设

建议至少建设四类数据集：

1. 语音识别集
2. 对话理解集
3. 任务执行集
4. 语音回复集

覆盖场景包括：

- 安静/噪声环境
- 远场/近场
- 快速说话
- 口音
- 中英混说
- 多语言
- 单轮/多轮
- 确认/否认
- 缺槽位补问
- 长答复口播压缩

### 14.2 线上埋点

建议埋点以下关键事件：

- `speech_started`
- `speech_ended`
- `vad_triggered`
- `asr_partial_received`
- `asr_final_received`
- `router_decided`
- `llm_started`
- `first_token_received`
- `tool_called`
- `tool_finished`
- `tts_started`
- `tts_first_audio`
- `tts_interrupted`
- `user_interrupted`
- `task_completed`
- `task_failed`

### 14.3 失败归因

建议建立标准错误分类：

- VAD 错误
- ASR 错误
- Router 错误
- LLM 理解错误
- Tool 参数错误
- Tool 执行失败
- 会话状态错误
- TTS 口播问题

### 14.4 迭代顺序建议

优先修复：

- 首响应慢
- ASR 关键实体错误
- VAD 截断
- 工具调用失败
- 打断不生效
- 任务完成率低

再优化：

- 追问策略
- 口播长度
- 占位反馈
- 风格自然度
- 多语言本地化表达

### 14.5 建议的优化闭环

```text
线上日志与样本采集
    ->
失败归因
    ->
选取 Top 问题
    ->
针对性调整（VAD / ASR / Router / Prompt / Tool / Response Planner）
    ->
离线回归测试
    ->
小流量 A/B
    ->
全量上线
    ->
样本回灌
```

## 17. 风险与注意事项

### 17.1 最容易踩的坑

- 过早追求全双工、全能 Agent、多语言全覆盖
- 把核心设计成聊天框后端，导致后续难以复用
- 工具执行缺乏确认机制
- 内部执行状态与用户可见状态混在一起
- 会话状态和任务状态未分离
- 只关注模型，不关注端到端工程链路

### 17.2 关键设计原则

- 文本核心做成事件驱动 Runtime，而不是单纯文本问答
- 任务状态与交互状态分离
- 内部执行状态与用户可见状态分离
- Action 和结构化结果优先，不要只输出长文本
- 语音优先 summary，文本支持 details
- 小模型优先、置信度不足自动升级
- 评估一定覆盖完整链路

---

## 18. 总结

本项目的关键并不只是“接入一个 ASR、一个 TTS、一个 LLM”，而是构建一个可跨渠道复用的智能运行时，并在其上叠加语音特有的实时交互系统。

在工程实现上，最稳妥的路径是：

- 把文本智能核心抽象成事件驱动的 Conversation Runtime
- 将语音部分独立为 Voice Runtime 和 Session Manager
- 用 Action、Task State、Presentation Layer 解耦文本与语音差异
- 用 NLP/规则 + 小模型 + 重模型 + Agent 构建混合路由
- 用整链路评测和回放归因来持续优化体验

如果边界设计正确，该架构不仅能支撑语音机器人，也能支撑文本机器人、电话机器人、企业 IM 助手和车载助手等多种渠道形态。
